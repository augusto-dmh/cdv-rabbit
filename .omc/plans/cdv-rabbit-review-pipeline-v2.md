### 6.1 Auto-merge interaction (CRITICAL â€” surfaced post-grilling)

Target review consumer (DocInt) has **auto-merge enabled at the repo level**. With v2 p95 latency at ~75s, a review that fires on `pullrequest:created` lands *after* the merge unless one of the following gates exists:

1. **Status-check gate (preferred).** cdv-rabbit posts a `pending` commit status on the PR head SHA immediately on webhook receipt; DocInt's branch protection requires the `cdv-rabbit/review` status as `success` before auto-merge fires. Implementation: `CommentPoster::postPendingStatus()` called at the top of `ReviewPullRequestJob::handle()`; `postFinalStatus()` at the bottom. This is the only design that preserves v2's value as a *gating* reviewer rather than a *post-mortem* annotator.
2. **Accept post-merge reviews (degraded value).** Reviews still post inline + summary, but on already-merged PRs. Useful as a learning/audit trail; useless as a quality gate. Reviewers see the comments after the fact.
3. **SCM-level wait config.** Bitbucket has no native "wait for external check before auto-merge" beyond branch-level required-status-checks. GitHub does (`required_status_checks` on branch protection) â€” leverage it.

**Decision required before W7-T5 rollout (NOT a code decision; an operational one):** which gate does the pilot DocInt repo use? Default recommendation: status-check gate (option 1) â€” codify `cdv-rabbit/review` as a required status check on the target repo's main branch; require pilot consumers to add it to their branch protection rules. v2 ships the status posting capability unconditionally; whether consumers enforce it is per-consumer.

New AC51 (added below): cdv-rabbit posts `pending` then `success`/`failure` status on every reviewed PR, recorded in `reviews.status_check_state`, so consumers can opt into status-check-gated auto-merge.

## 8. Rollback

Per-Workspace, zero-migration rollback:
1. Admin PATCH `/workspaces/{slug}` with `review_schema_version=v1`.
2. Next `ReviewPullRequestJob::handle()` reads `v1`, takes the v1 path.
3. In-flight v2 reviews complete normally (no abort).
4. v1 frozen artifacts (`review_v1.txt`, `review_result_v1.json`) are always present.
5. Global rollback: set `config('cdv-rabbit.review_schema_version_default')` to `v1` and run a bulk update on workspaces that opted into v2.
