# Eval harness v2 coverage gap and the recall-zero signal on golden fixtures

Real-world validation of the Agent Prompt feature (`ec4a3ac`, ADR 0006) surfaced two pre-existing gaps in the `rabbit:eval` harness that the v1.2 reconciliation never closed. First, `App\Console\Commands\EvalCommand` rejects `--schema=v2` outright (returns exit 2 with `"schema v2 not yet implemented (W7-T2 scope)"`) even though Phase 7 shipped the v2 schema, prompt, and pipeline; the locked contract in `AGENTS.md` §4 (*"v2 prompt/schema gated by eval baseline"*) and AC50 (*"v2 prompt changes do not merge without delta vs golden baseline"*) are declared but **not enforceable** because the harness path they reference does not exist. Second, `LlmJudge::ROTATION` is a hard-coded `anthropic <-> openai` swap with no same-provider fallback (AC41 makes this deliberate — cross-provider rotation eliminates same-family sycophancy), so an environment that only carries one provider's credentials cannot run the eval at all. A local-only override that swapped the rotation to `openai => openai` was prototyped and reverted; the right path is an explicit `same-provider` config flag fronted by a loud warning, not silent bypass.

A standalone v2 mini-eval (`/tmp/mini-eval-v2.php`, not committed) was run against the 10-fixture golden corpus using `OpenAiReviewer::reviewDiffV2()` + `critiqueDraft()` directly, sidestepping `rabbit:eval`. Five fixtures completed (the other five hit OpenAI rate limits without retry backoff); every successful cell emitted **zero approved Findings**, against ground-truth counts of 3-6 per fixture. The path-aware naive matcher (path + severity overlap) produced recall = 0.000 across the board. This is recall = 0 on the golden corpus through the OpenAI v2 path, well below the AC39 / AC40 threshold of 0.7. The same pipeline against a live PR (DocInt #39, the `feat-archivable` smoke target) produced 2 inline Findings — both rendered with correctly-formed `agent_prompt` blocks — out of approximately 6 plantings. The two signals together suggest the v2 OpenAI path has a recall problem that mini-eval surfaces dramatically (perhaps amplified by feeding the unchunked diff directly when production chunks per file) but live runs still surface some signal. The Anthropic v2 path was not exercised because no `ANTHROPIC_API_KEY` is present in this environment.

The Agent Prompt feature (ADR 0006) is **not implicated** in the recall gap. The `findings[].agent_prompt` field is strictly additive: the LLM populates it alongside `message` and `suggestion`, the critic does not verdict it (anchor-only contract preserved), and `CommentPoster::renderAgentPromptBlock()` only fires when a Finding already exists. A Finding-count drop cannot be a side effect of adding a nullable field to the schema; the few-shot updates in `review_v2.txt` (5 positive exemplars gained an `agent_prompt` line, `Forbidden content` was amended to allow `@<path>` inside `agent_prompt`) do not relax the `Findings vs Nitpicks` discipline. The `CommentSanitizer` regex change (preserve `@<path>`, strip `@<username>`) is also additive — anything the v1 sanitizer let through, the v2 regex lets through. Live PR #39 validates this directly: AC52 (footer rendered), AC53 (path preserved), AC54 (strip+rehydrate worked), AC56 (no `<details>` for null prompt) all green on real GitHub comments.

Action items, ranked by recovery value:

1. **Add `--schema=v2` support to `App\Console\Commands\EvalCommand`** so the locked contract in `AGENTS.md` §4 is actually enforceable. Wire `evaluateCell()` to call `reviewDiffV2()` + `critiqueDraft()` and apply the approved-findings filter before judging. Mirrors the production `ReviewPullRequestJob::handle()` v2 branch.

2. **Add a `same-provider` rotation flag to `LlmJudge`** (e.g. `config('cdv-rabbit.eval.cross_provider_judge', true)`) so single-provider environments can run the eval — explicitly opt-out, with a startup warning in the eval command output noting the loss of cross-family bias control. AC41 stays the default; the flag is for unblock-only contexts.

3. **Investigate the v2 OpenAI recall regression directly.** Run the production-shaped pipeline (chunked per file, full `ReviewPullRequestJob` orchestration) against a single golden fixture and compare with `OpenAiReviewer::reviewDiffV2()` called on the whole diff. If chunking is the recall driver, document chunking as a v2 hard requirement and update `review-pipeline-v2.md` §5.2 to record the constraint. If chunking is not the driver, the `review_v2.txt` exemplar set or the strict schema is suppressing Findings on smaller diffs and the prompt needs tuning — both before the v2 rollout reaches more Workspaces.

4. **Backfill the corpus with at least one Bitbucket fixture** so the eval harness reflects production reality (today's 10 golden PRs are GitHub-only via DocInt/cdv-rabbit/intranet-cdv).

Until item 1 lands, the `golden-pr-eval` CI gate referenced in `AGENTS.md` §4 is **non-load-bearing** — it can never block a v2 prompt or schema change. The merge gate is functionally live PR smoke + this ADR's transparency entry, not the eval baseline. This is the same disclaimer the v1.2 reconciliation should have carried but did not. The Agent Prompt feature merges with that constraint stated openly rather than masked.
