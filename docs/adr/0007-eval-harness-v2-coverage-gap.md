# Eval harness v2 coverage gap and the recall-zero signal on golden fixtures

Real-world validation of the Agent Prompt feature (`ec4a3ac`, ADR 0006) surfaced two pre-existing gaps in the `rabbit:eval` harness that the v1.2 reconciliation never closed. First, `App\Console\Commands\EvalCommand` rejects `--schema=v2` outright (returns exit 2 with `"schema v2 not yet implemented (W7-T2 scope)"`) even though Phase 7 shipped the v2 schema, prompt, and pipeline; the locked contract in `AGENTS.md` §4 (*"v2 prompt/schema gated by eval baseline"*) and AC50 (*"v2 prompt changes do not merge without delta vs golden baseline"*) are declared but **not enforceable** because the harness path they reference does not exist. Second, `LlmJudge::ROTATION` is a hard-coded `anthropic <-> openai` swap with no same-provider fallback (AC41 makes this deliberate — cross-provider rotation eliminates same-family sycophancy), so an environment that only carries one provider's credentials cannot run the eval at all. A local-only override that swapped the rotation to `openai => openai` was prototyped and reverted; the right path is an explicit `same-provider` config flag fronted by a loud warning, not silent bypass.

A standalone v2 mini-eval (`/tmp/mini-eval-v2.php`, not committed) was run against the 10-fixture golden corpus using `OpenAiReviewer::reviewDiffV2()` + `critiqueDraft()` directly, sidestepping `rabbit:eval`. Five fixtures completed (the other five hit OpenAI rate limits without retry backoff); every successful cell emitted **zero approved Findings**, against ground-truth counts of 3-6 per fixture. The path-aware naive matcher (path + severity overlap) produced recall = 0.000 across the board. This is recall = 0 on the golden corpus through the OpenAI v2 path, well below the AC39 / AC40 threshold of 0.7. The same pipeline against a live PR (DocInt #39, the `feat-archivable` smoke target) produced 2 inline Findings — both rendered with correctly-formed `agent_prompt` blocks — out of approximately 6 plantings. The two signals together suggest the v2 OpenAI path has a recall problem that mini-eval surfaces dramatically (perhaps amplified by feeding the unchunked diff directly when production chunks per file) but live runs still surface some signal. The Anthropic v2 path was not exercised because no `ANTHROPIC_API_KEY` is present in this environment.

The Agent Prompt feature (ADR 0006) is **not implicated** in the recall gap. The `findings[].agent_prompt` field is strictly additive: the LLM populates it alongside `message` and `suggestion`, the critic does not verdict it (anchor-only contract preserved), and `CommentPoster::renderAgentPromptBlock()` only fires when a Finding already exists. A Finding-count drop cannot be a side effect of adding a nullable field to the schema; the few-shot updates in `review_v2.txt` (5 positive exemplars gained an `agent_prompt` line, `Forbidden content` was amended to allow `@<path>` inside `agent_prompt`) do not relax the `Findings vs Nitpicks` discipline. The `CommentSanitizer` regex change (preserve `@<path>`, strip `@<username>`) is also additive — anything the v1 sanitizer let through, the v2 regex lets through. Live PR #39 validates this directly: AC52 (footer rendered), AC53 (path preserved), AC54 (strip+rehydrate worked), AC56 (no `<details>` for null prompt) all green on real GitHub comments.

Action items, ranked by recovery value:

1. **Add `--schema=v2` support to `App\Console\Commands\EvalCommand`** so the locked contract in `AGENTS.md` §4 is actually enforceable. Wire `evaluateCell()` to call `reviewDiffV2()` + `critiqueDraft()` and apply the approved-findings filter before judging. Mirrors the production `ReviewPullRequestJob::handle()` v2 branch.

2. **Add a `same-provider` rotation flag to `LlmJudge`** (e.g. `config('cdv-rabbit.eval.cross_provider_judge', true)`) so single-provider environments can run the eval — explicitly opt-out, with a startup warning in the eval command output noting the loss of cross-family bias control. AC41 stays the default; the flag is for unblock-only contexts.

3. **Investigate the v2 OpenAI recall regression directly.** Run the production-shaped pipeline (chunked per file, full `ReviewPullRequestJob` orchestration) against a single golden fixture and compare with `OpenAiReviewer::reviewDiffV2()` called on the whole diff. If chunking is the recall driver, document chunking as a v2 hard requirement and update `review-pipeline-v2.md` §5.2 to record the constraint. If chunking is not the driver, the `review_v2.txt` exemplar set or the strict schema is suppressing Findings on smaller diffs and the prompt needs tuning — both before the v2 rollout reaches more Workspaces.

4. **Backfill the corpus with at least one Bitbucket fixture** so the eval harness reflects production reality (today's 10 golden PRs are GitHub-only via DocInt/cdv-rabbit/intranet-cdv).

Until item 1 lands, the `golden-pr-eval` CI gate referenced in `AGENTS.md` §4 is **non-load-bearing** — it can never block a v2 prompt or schema change. The merge gate is functionally live PR smoke + this ADR's transparency entry, not the eval baseline. This is the same disclaimer the v1.2 reconciliation should have carried but did not. The Agent Prompt feature merges with that constraint stated openly rather than masked.

## 2026-05-17 follow-up — items 1 and 2 closed, item 3 elevated

Commit `23deb3b` lands items 1 (wire `--schema=v2` into `EvalCommand`) and 2 (`cdv-rabbit.eval.cross_provider_judge` flag in `LlmJudge`). `rabbit:eval --schema=v2 --provider=openai` now executes the production-shaped draft → critique pipeline, filters to critic-approved Findings before judging, and `LlmJudge` falls back to same-provider when the cross-provider flag is off. 475 tests / 473 passed / 2 skipped / 0 failed; pint clean.

Running the now-functional eval immediately surfaced two concrete bugs that block item 3 (recall investigation) and elevate it from "investigate slowly" to "fix before any v2 quality baseline can be recorded":

**Bug A — `OpenAiReviewAgent: schema must be set before generating.`** All 6 cells that did not rate-limit failed at the `OpenAiReviewer::reviewDiffV2()` step with this exact error. The fluent chain `(new OpenAiReviewAgent)->withInstructions(...)->withSchema(...)->prompt(...)` sets `$jsonSchema` on the local instance, but somewhere in `laravel/ai`'s `Promptable::prompt()` → `Provider::prompt()` → gateway pipeline, the agent that is asked for its `schema()` is a *different* instance — one with `$jsonSchema = []` from the constructor default — which throws on line 78 of `OpenAiReviewAgent`. Live PR #39 succeeded against the same code path; the strongest hypothesis is that the live `ReviewPullRequestJob::runV2Pipeline()` runs in a request/job context where the container resolves the agent before fluent state is set up, while the CLI `rabbit:eval` path lands on a different resolution branch. The eval cells that ran second-to-last in the loop sometimes succeeded and sometimes failed, suggesting container state leak between iterations. Fix path: either rebuild `OpenAiReviewer::reviewDiffV2()` around `OpenAiReviewAgent::make()->withSchema(...)` (the Promptable static factory that goes through `Container::makeWith`) or stop relying on fluent state and pass the schema through the `AgentPrompt` constructor explicitly.

**Bug B — OpenAI rate limiting on sequential cells.** 4 of 10 cells terminated with `Application rate limited by AI provider [openai]`. The eval issues one `gpt-4o` call per fixture sequentially (no concurrency), and the typical `gpt-4o` Tier-1 quota is 500 RPM — well above the 10-cell load. The trigger is probably the `Responses API` structured-output endpoint sharing a tighter sub-quota OR the failure mode of Bug A producing fast-retry storms inside `laravel/ai`'s failover loop. The eval should ship a small inter-cell delay (`sleep(2)`) and the `OpenAiErrorClassifier` should map the 429 to a non-retryable terminal so the eval reports the cell as ERROR rather than chewing through quota on retries.

Until A is fixed, the v2 quality baseline from `rabbit:eval` is still not obtainable. The Agent Prompt feature's live signal (PR #39 produced 2 well-formed Findings with rendered `agent_prompt` blocks) is the only end-to-end v2 validation on record. Items 3 and 4 stay open; the new agenda is:

5. **Fix Bug A** — refactor `OpenAiReviewer::reviewDiffV2()` and `critiqueDraft()` to pass schema through `AgentPrompt` instead of relying on `OpenAiReviewAgent` fluent state surviving the Promptable resolution.
6. **Fix Bug B** — sleep 2s between cells in `EvalCommand::handle()` and tag 429 as terminal in `OpenAiErrorClassifier` so retry storms die fast.

The locked contract in `AGENTS.md` §4 remains **declared but non-load-bearing** until 5 and 6 land. Item 1's plumbing is correct; the v2 path simply does not survive contact with `laravel/ai` outside the request/queue runtime.

## 2026-05-17 follow-up (second pass) — Bug A closed, Bug B partially mitigated, item 3 confirmed empirically

Item 5 (Bug A) turned out to be much narrower than the original hypothesis. The fluent state on `OpenAiReviewAgent` was fine — the bug was that `LaravelAiJudgeLlmCaller::callOpenAi()` reused `OpenAiReviewAgent` (which implements `HasStructuredOutput`) without calling `withSchema()`, so the laravel/ai gateway happily invoked `schema()` on the agent and tripped on the empty array. The judge prompt does NOT want a structured output — it wants plain-text JSON the caller parses. Fix: a dedicated `App\Ai\Agents\OpenAiJudgeAgent` that implements `Agent` (and uses `Promptable`) but does **not** implement `HasStructuredOutput`. `LaravelAiJudgeLlmCaller` now constructs `OpenAiJudgeAgent` for the OpenAI cell. The original instrumented stack trace tells the whole story — the second `schema()` call had `systemInstructions` starting with `"You are an impartial code-review-quality"` (the judge prompt), not `"You are an expert code reviewer integrat"` (the review prompt). Two different agents, same class, schema only set on the first one. Reviewer-side fluent state was the red herring.

Item 6 (Bug B) shipped as a `--cell-delay` CLI option (default 3s, configurable per run) plus a no-op of the original "tag 429 terminal" idea — the OpenAi classifier already routes 429 to `RetryWithBackoff`, which is the right behaviour for production; making it terminal would weaken `ReviewPullRequestJob`. Inter-cell delay covers the eval case without touching production. With `--cell-delay=5`, 6/10 cells complete (vs 4/10 without delay); the 4 cells that still rate-limit are all the DocInt fixtures — large diffs that spend more TPM per call. A longer delay (8-10s) likely closes the gap; for CI we'd recommend `--cell-delay=10` plus retry-once-on-error on top.

Item 3 (recall regression) is now confirmed empirically rather than hypothesised. Across the 6 fixtures that completed the same-provider OpenAI v2 eval with cell-delay=5: every single cell returned **recall = 0**, with one fixture (`cdv-rabbit-19d2df8`) emitting 1 hallucinated Finding (precision = 0) and the remaining 5 emitting 0 Findings at all (recall = 0, precision = 1 by convention since there are no false positives). The live PR #39 path produced 2 well-formed Findings against a real diff, so v2 is not broken — the divergence is between the eval shape (whole diff fed to one call) and the production shape (per-file chunking via `ReviewPullRequestJob::runV2Pipeline()`). The v2 prompt and few-shot exemplars are tuned for small per-file inputs; given a multi-file diff the LLM emits little to nothing. Closing this requires either chunking inside the eval to match production, or extending the v2 prompt with multi-file exemplars and retuning the critic threshold for aggregated drafts. Either change must re-pay the golden-baseline gate it itself is supposed to enforce.

Updated agenda:

5. ✅ Closed (e028127 → this follow-up). `OpenAiJudgeAgent` lives in `app/Ai/Agents/`; `LaravelAiJudgeLlmCaller::callOpenAi()` switched to it; review and critic agents untouched.
6. 🟡 Partially mitigated. `--cell-delay` option lands; DocInt fixtures still TPM-blocked at 5s. Bump default to 8-10 if CI keeps failing.
3. 🔁 Confirmed empirically — v2 eval recall = 0 on 6/6 completing OpenAI cells. Next step is reshaping the eval to match production chunking, OR retuning the v2 prompt for whole-diff inputs.

The `golden-pr-eval` gate is now technically *runnable* with `--schema=v2 --provider=openai --cell-delay=10` and `CDV_RABBIT_EVAL_CROSS_PROVIDER_JUDGE=false`, but the recall floor it would enforce (`>= 0.7`) is two orders of magnitude above what the path currently delivers. Until item 3 is closed, the gate stays advisory rather than blocking in `AGENTS.md` §4 (caveat from v1.4 still stands). The Agent Prompt feature continues to land safely under that disclosure: it is strictly additive, validated end-to-end against a real PR, and cannot mask a recall regression that pre-dates it.

## §7 — Error classifier mis-classifies SDK-wrapped 429 as Terminal

**Context**: Discovered during PR #41 review (2026-05-17). The three cells that hit `gpt-4o` Tier-1 TPM exhaustion were all recorded as Terminal failures rather than being retried — despite `error.code = rate_limit_exceeded` in the OpenAI response body. Root cause: `laravel/ai`'s `HandlesFailoverErrors::withErrorHandling()` catches `Illuminate\Http\Client\RequestException` on 429 and re-raises it as `Laravel\Ai\Exceptions\RateLimitedException` (passing the original exception as `$previous`) before `OpenAiErrorClassifier` sees it. The classifier's `resolveHttpStatus` and `resolveErrorCode` methods only walked `GuzzleRequestException` nodes in the previous-chain — they skipped `LaravelRequestException` nodes inside the chain — so the body with `rate_limit_exceeded` was never read, status resolved to `null`, and the exception fell through to the `Terminal` default. **Decision**: both `resolveHttpStatus` and `resolveErrorCode` now check for `LaravelRequestException` at every node of the `getPrevious()` walk (not only at the top level), so the body and status are readable regardless of wrapping depth. Additionally, if the entire chain is exhausted without resolving either field but the top-level exception is `RateLimitedException`, the classifier returns `RetryWithBackoff` as a safe conservative default — the SDK only raises that type for genuine rate-limit conditions. `insufficient_quota` (also a 429 but with a distinct `error.code`) still resolves to `Terminal` through the body-read path. **Consequence**: transient TPM-exhaustion errors now retry with backoff inside `ReviewPullRequestJob` instead of failing the job immediately; `insufficient_quota` continues to terminate as before; the no-previous-chain edge case (SDK raises without a response object) is handled conservatively rather than silently. **Status**: Closed. Changes in `app/Support/OpenAiErrorClassifier.php`; 21 new Pest assertions in `tests/Unit/Support/OpenAiErrorClassifierTest.php` (all green).
