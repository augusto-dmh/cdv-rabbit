# Per-provider webhook controllers and asymmetric HMAC scheme

Webhook HMAC verification is asymmetric between SCM providers: Bitbucket uses a per-Workspace `webhook_secret` registered with the BB API at register-webhook time (plus a defense-in-depth URL token per repo); GitHub uses a per-App secret set on the GitHub App registration (no URL token, no per-repo registration — the App emits webhooks to its fixed URL for every installed repo). Because verification surfaces, payload shapes, event-name conventions, and idempotency-id headers (`X-Hook-UUID` vs `X-GitHub-Delivery`) all differ, each provider gets its own controller (`Bitbucket/WebhookController`, `Github/WebhookController`) sharing a small `WebhookIngestionPipeline` for the post-validation steps (delivery insert + job dispatch). The `AGENTS.md §4` "HMAC webhook auth" locked contract is rephrased to make the asymmetry explicit. The `webhook_deliveries.bitbucket_uuid` column is renamed to `scm_delivery_id` to drop the provider-specific name. Trade-off: a unified controller with `switch ($provider)` was rejected because branching inside HMAC verification is a regression-prone footgun on a security-critical path; the small duplication is worth the linear, auditable verification code per provider.
